# Working Note: Python Parser Strategy Frontier

This artifact defines the current Python parser-deepening decision boundary after the delivered regex-based decorator/class-scope tranche.

## Current Evidence

What is already delivered:

- imported-symbol resolution
- relative-import normalization
- module-alias handling
- `self` / `cls` ownership
- decorated `@classmethod` / `@staticmethod` ownership
- immediate-scope nested local suppression
- conservative `@property` behavior
- Python test-file linkage

What is not yet evidenced as an acute repo-level failure:

- benchmark or replay artifacts proving that Python retrieval quality is currently blocked by the regex/stateful parser
- a dedicated Python frontier note that turns “deeper parsing” into a bounded implementation slice

What remains a plausible parser ceiling:

- multiline defs/imports
- richer nested scopes
- more complex decorator stacks
- newer syntax surfaces where line-state parsing becomes increasingly heuristic-heavy

## Decision

Current outcome:

`defer deeper parsing for now`

Reason:

- `ADR-028` still correctly identifies Python as the next structural-parser decision after Elixir
- however, the current repository evidence does not yet show a concrete Python retrieval failure set that justifies opening an implementation tranche immediately
- the next Python step should therefore be an evidence-gathering and strategy-selection slice, not a parser implementation commitment

## Required Preconditions To Reopen Python Parser Work

Open a Python parser-deepening tranche only when at least one of the following is true:

- benchmark fixtures or replay cases show Python retrieval misses that are best explained by parser-shape limits
- new supported Python surfaces require parsing that the current regex/stateful approach cannot keep local and explainable
- maintainers explicitly prioritize Python over interprocedural/dataflow work despite the current evidence gap

## Preferred Strategy Once Reopened

Preferred first comparison:

- `native Python AST`
- versus `tree-sitter`

Default preference today:

`native Python AST preferred for the first serious deepening slice`

Why:

- Python already has a strong language-native structural parser
- the repository need is structural correctness, not tree-sitter symmetry
- the first Python deepening slice should choose the simplest parser that improves correctness while preserving current runtime contracts

## Bounded First Slice If Reopened

If Python parser work is reopened, the first implementation slice should stay additive and avoid a default flip.

Recommended first slice:

- add a structural parser path behind explicit parser selection
- preserve the current Python regex/stateful parser as fallback
- target only documented failure surfaces proved by benchmark/replay evidence
- keep `parse-file` output shape and diagnostics stable

Acceptance for a future implementation tranche:

- targeted Python benchmark or replay cases prove improvement on the motivating failures
- `clojure -M:test` remains green
- no public confidence-ceiling bump in the same tranche

## Done Definition For This Note

This note is complete for now because it answers:

- why Python is not being implemented next by default
- what evidence must exist before reopening parser work
- which parser strategy is preferred once the work is reopened

The next Python artifact should be either:

- a benchmark/replay evidence note proving the need to reopen parser work, or
- a Python implementation-slice note once that evidence exists
