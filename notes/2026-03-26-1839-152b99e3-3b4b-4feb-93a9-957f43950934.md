# Working Note: Interprocedural and Dataflow Frontier

This artifact turns the repo’s current “next frontier” language into a bounded first tranche.

## Problem Statement

The repository now has stronger intra-unit semantic extraction across its language lanes, but it still stops short of compiler-grade interprocedural reasoning.

The next tranche should improve retrieval and ranking using conservative cross-unit flow facts without opening a whole-program analysis project.

## V1 Decision

Current v1 scope:

- one-hop interprocedural forwarding only
- conservative argument/return propagation inside a module or directly linked file boundary
- retrieval-facing improvements only where evidence is replayable
- no fixed-point global analysis
- no whole-program call graph saturation
- no language-wide compiler emulation

## First Participating Surfaces

Start with surfaces that already have strong unit identity and caller linkage:

- Clojure helper forwarding patterns already covered by semantic-core tests
- Java inherited/helper-mediated call chains where ownership is already explicit
- Elixir helper forwarding only after parser/readiness work is stable

Do not start with Python or Lua in the first interprocedural slice.

Reason:

- the first tranche should attach to lanes with stronger existing ownership signals rather than stacking parser uncertainty and interprocedural uncertainty together

## Boundary Model

Policy boundary:

- interprocedural evidence rules decide when a forwarded relationship is safe enough to expose to retrieval

Extraction boundary:

- language lanes continue to emit local semantic facts
- the interprocedural layer consumes those facts; it does not ask language parsers to become whole-program engines

Evidence boundary:

- benchmark/replay fixtures own acceptance
- new forwarded relationships are accepted only when they improve known retrieval scenarios without creating cross-language confidence inflation

## First Supported Flow Shapes

The first implementation slice should support only:

- one-hop helper forwarding where callee identity is already known
- simple wrapper functions that pass through one primary argument
- direct return-of-call forwarding where the local unit acts as a thin adapter

Explicitly out of scope for v1:

- branch-sensitive dataflow
- multi-hop propagation
- alias-heavy whole-module propagation
- mutation/state tracking
- container/collection flow modeling
- cross-service or transport-level tracing

## Acceptance Criteria For A Future Implementation Tranche

Before implementation starts, the tranche must define replayable fixtures proving that interprocedural facts improve retrieval.

A v1 implementation is done only when:

- targeted benchmark or replay cases exist for each supported flow shape
- improvements are observable in ranking, authority units, caller targeting, or related-test linkage
- `clojure -M:test` remains green
- no public confidence-ceiling bump ships in the same tranche

## Done Definition For This Note

This note is complete for now because it locks:

- the v1 scope boundary
- the first participating lanes
- the flow shapes that count as “interprocedural” in the first slice
- the acceptance standard for opening implementation work

The next artifact should be an implementation-slice note once concrete replay or benchmark cases are assembled.
